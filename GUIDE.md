# מדריך שימוש — CCSO v3.0

## מה זה CCSO

CCSO הוא שכבת אופטימיזציה שמיועדת קודם כל ל-`Claude Code`.

העיקרון החשוב הוא:

- אם CCSO שולח את הפרומפט בעצמו, הוא יכול למדוד עלות וחיסכון.
- אם CCSO רק כותב קבצי חוקים או הוראות לכלי אחר, הוא יכול לעזור, אבל לא למדוד את הצ'אטים של אותו כלי.

לכן הדשבורד מציג חיסכון אמיתי רק במסלולים ש-CCSO באמת שולט בהם.

## התחלה מהירה

```bash
ccso
```

אם אתה רוצה לפתוח את הדשבורד:

```bash
ccso --dashboard
```

אם אתה רוצה לינק ציבורי אמיתי לדשבורד החי:

```bash
ccso --share-dashboard
```

זה לא deployment סטטי. זה tunnel אל הדשבורד המקומי שלך, ולכן הוא שומר את כל הפונקציות האמיתיות.

או בלחיצה כפולה מתוך התיקייה:

```bash
./הפעל\ CCSO.command
```

## מה קורה אוטומטית כברירת מחדל

### 1. ניתוב מודל חכם

CCSO בוחר מודל לפי סוג המשימה:

- שאלות פשוטות: `Haiku`
- משימות קוד רגילות: `Sonnet`
- משימות מורכבות: `Opus`

### 2. Cache שמרני

אם אותו פרומפט נשלח שוב באותו הקשר, CCSO יכול להחזיר תשובה מהמטמון ולחסוך פנייה מלאה למודל.

### 3. קיצוץ לוגים ממוקד

במקום לשלוח לוג ארוך בצורה עיוורת, CCSO שומר את אזור השגיאה וה-stack trace.

### 4. זיכרון רלוונטי בלבד

CCSO שומר עובדות קצרות על הפרויקט ומזריק רק מה שרלוונטי לפרומפט הנוכחי, תחת תקציב טוקנים מוגבל.

### 5. Session Reset Advisor

CCSO עוקב אחרי מספר turns, זמן רצוף וטוקנים מצטברים.

כשהסשן מתחיל להתנפח, הוא לא "ממציא חיסכון", אלא פשוט מציע לעשות `handoff` או לפתוח סשן חדש כדי לשמור על איכות וקונטקסט נקי.

### 6. Reusable File Briefs

אפשר לשמור brief קצר לקובץ שחוזר שוב ושוב, ואז CCSO יזריק אותו רק כשאותו קובץ מוזכר.

זה נועד לחסוך הדבקה חוזרת של קונטקסט, אבל הדשבורד מציג את זה כ-`Estimated context reuse` ולא כחיסכון קשיח.

### 7. Prompt Bundler

אם יש לך כמה משימות קשורות, אפשר לאגד אותן לבקשה אחת עם `/bundle`.

זה מפחית הלוך-חזור ושימוש חוזר באותו רקע, אבל שוב: זה כלי workflow, לא מדד חיסכון ישיר.

## מה כבוי כברירת מחדל

פיצ'רים שעלולים לפגוע בדיוק או לשנות ניסוח נשארים כבויים עד שתדליק אותם:

- `translate`
- `codeCompression`
- `truncateLargePastes`
- `dedupeLongInput`
- `responseLengthHints`

זה נעשה בכוונה כדי לשמור על איכות ולא "לרדוף אחרי חיסכון" בכל מחיר.

## פקודות בתוך ה-REPL

| פקודה | מה עושה |
|---|---|
| `/status` | הצגת עלות, פקודות, זמן ומצב נוכחי |
| `/handoff` | שמירת סיכום סשן ואיפוס |
| `/cache` | הצגת מצב המטמון |
| `/cache clear` | ניקוי המטמון |
| `/history` | הצגת היסטוריית פרומפטים |
| `/memory list` | הצגת זיכרון שמור |
| `/memory add "עובדה"` | שמירת עובדה על הפרויקט |
| `/memory clear` | ניקוי זיכרון הפרויקט |
| `/memory global "עובדה"` | שמירת עובדה גלובלית |
| `/brief list` | הצגת briefs שמורים |
| `/brief save <path>` | שמירת brief קצר לקובץ |
| `/brief show <path>` | הצגת brief שמור לקובץ |
| `/brief clear` | מחיקת briefs שמורים |
| `/bundle ...` | איגוד כמה סעיפים לבקשה אחת |
| `/template` | הצגת תבניות מוכנות |
| `/dashboard` | פתיחת הדשבורד |
| `/exit` | יציאה |

## הדשבורד

הדשבורד רץ בכתובת:

```text
http://localhost:3847
```

מה תראה שם:

- `Spend` שנמדד בפועל
- `Net savings` כאומדן שקוף לפי breakdown
- כרטיסי מצב לאופטימיזציות
- גרפים יומיים
- פירוט פלטפורמות לפי רמת תמיכה
- צ'אט דשבורד

### איך לקרוא את המספרים נכון

- `Measured runtime` אומר ש-CCSO באמת שולח את הפרומפט ולכן יכול למדוד.
- `Assisted, not measured` אומר ש-CCSO עוזר לכלי, אבל לא רואה את הצ'אט שלו.
- `Instruction file only` או `Project config only` אומר שיש תמיכה בקבצים, לא בזמן ריצה.
- `Estimated context reuse` אומר שיש פיצ'רים כמו memory ו-briefs שעוזרים לא לחזור על אותו הקשר, אבל CCSO לא יכול לדעת בוודאות מה היית מדביק אחרת.
- `Quota / session hygiene` אומר שיש עצות כמו handoff ו-reset advisor שמונעות סשנים מנופחים, בלי להיחשב כחיסכון ישיר.

## מתי באמת תראה חיסכון

תראה חיסכון בדשבורד רק אם:

1. שלחת פרומפטים דרך CCSO עצמו.
2. הופעלו אופטימיזציות אמיתיות כמו routing, cache, trimming, memory.
3. נוצר מספיק שימוש כדי שיהיה מה למדוד.

אם עבדת שעות בתוך Cursor בלבד, זה לא יופיע כחיסכון בדשבורד, כי CCSO לא רואה את הצ'אטים הפנימיים של Cursor.

## שינוי הגדרות

```bash
ccso --config
```

כאן אפשר להדליק או לכבות פיצ'רים, כולל lossy features במידת הצורך.

## טיפ לשימוש נכון

הזרימה המומלצת היא:

1. לעבוד דרך `ccso` כשאתה רוצה חיסכון ומדידה אמיתיים.
2. להשתמש ב-`inject` כדי לשפר כלים אחרים בלי להעמיד פנים שיש שם מדידה מלאה.
3. להשתמש ב-`/brief` לקבצים שחוזרים שוב ושוב וב-`/bundle` כשיש כמה שאלות קשורות.
4. להדליק lossy features רק כשיש לך סיבה ברורה.
